# Proposal: Лаборатория, Фонтан жизни, Зомби-режим (v1)

## Статус

Draft v1.0 · расширяет [ADR 002](../decisions/002-timer-lab-fountain-nodes-bot-miniapp.md). Область: core loop, anti-tilt, базовый таймер и его история, экономика воскрешения зомби. Монетизация упомянута только как зависимость, её параметры — отдельно.

## 0. Контекст и тупики, которые закрываем

Оригинальный дизайн v1 даёт три места, где игрок теряет мотивацию:

1. **Контакты кончились** — таймер идёт, сделать нечего.
2. **Таймер истёк** — мгновенное превращение в зомби воспринимается как наказание.
3. **Зомби-режим** — непонятная полу-жизнь без нарратива и пути выхода.

Фикс — не убирать последствия, а дать **альтернативный путь действия** в любой момент:

- **Лаборатория** — автономный личный путь продления времени.
- **Фонтан жизни** — коллективный глобальный ивент с наградой и воскрешением.
- **Зомби как отдельная фаза** — полноценный режим со своими правилами.

Базовый горизонт таймера переосмысливается (суточный → 48–72 ч) вместе с видимой «историей роста», чтобы увеличение времени ощущалось **заработанным**.

## 1. Глоссарий (спек → код)

Фиксируем термины заранее, чтобы AC читались под фактическую схему.

| Термин в спеке | Имя в коде / БД (сейчас) | Примечание |
|---|---|---|
| Состояние Infected | `users.state = "infected"` | Основное активное состояние |
| Состояние Zombie | `users.state = "zombie"` | После истечения таймера |
| Состояние Immune | `users.state = "immune"` | После воскрешения (временное окно) |
| Таймер кризиса (expires_at) | `users.timer_ends_at` | TIMESTAMPTZ; у зомби — игнорируется (см. AC-ZMB-03) |
| Окно иммунитета | `users.immune_ends_at` | TIMESTAMPTZ |
| Базовый горизонт таймера | `settings.default_timer_hours` / `launch_timer_hours` | `app/config.py` |
| Бонус за реферал | `settings.bonus_hours_per_referral` | `app/config.py` |
| Продление во «сне» | `settings.sleeping_no_spread_hours`, `sleeping_tick_bonus_seconds` | Серверное; заменяется Лабораторией (см. §2) |
| Длительность иммунитета | `settings.immune_duration_hours` | Используется в `apply_immunity_recovery` |

**Отсутствующие состояния:** в модели нет `Human`. Везде в AC, где подразумевается «не играющий и не заражённый», следует читать «не-пользователь бота» или «пользователь вне `side="virus"`».

## 2. Базовый таймер

### 2.1 Параметры

Все числа — **рекомендации для конфига**, а не захардкоженные; итоговые значения утверждаются балансом.

| Параметр | v1 (код сейчас) | v2 (предложение) | Обоснование |
|---|---|---|---|
| `default_timer_hours` | 60 ч | 48 ч | Планетарный охват, часовые пояса |
| `launch_timer_hours` | 72 ч | 72 ч | Первые 7 дней после публичного запуска |
| `bonus_hours_per_referral` | 3 ч | 4 ч | Ребаланс под новый базовый горизонт |
| Прирост за Лабораторию | — | +4…+8 ч (RNG) | Новая механика |
| Прирост за Фонтан | — | +12 ч | Коллективная награда |
| Режим «сна» (`sleeping_*`) | активен | отключить | Лаборатория закрывает тот же кейс явно |

### 2.2 История роста таймера

Таймер в карточке показывается с раскрывающейся историей **последних 3 событий** роста, например:

```
⏱ 31:14:00  ДО РАСПАДА
  ├ +4 ч   реферал @masha (09:41)
  ├ +8 ч   Лаборатория — цикл завершён
  └ +12 ч  Фонтан #3 (вчера)
```

- Хранение: `users.timer_history JSONB` — массив из последних 3 событий `{delta_hours, reason, ts}`; старые вытесняются в FIFO.
- Обновление: при любом изменении `timer_ends_at` со стороны сервера (реферал, Лаба, Фонтан, ручное админ-действие).
- Событие «сон» из v1 при отключении механики больше не пишется.

## 3. Лаборатория

**Назначение:** автономный путь продления таймера без рефералов. Доступен всегда, даже при нулевом соцграфе.

### 3.1 Цикл (3 шага)

| Шаг | Название | Триггер | Задержка | Результат |
|---|---|---|---|---|
| 1 | Сбор образца | Кнопка в боте или Mini App | — | Образец создан, переход к шагу 2 |
| 2 | Анализ | Автоматически | 45 мин (серверный таймер в Redis) | Push «анализ готов» |
| 3 | Результат | Кнопка «Получить» | — | +4…+8 ч к таймеру **или** +1 Mutation Point (RNG) |

### 3.2 Детали шагов

**Шаг 1 — Сбор образца**

- Доступен только `state ∈ {infected, zombie}`.
- Одиночное нажатие; кнопка деактивируется до завершения цикла.

**Шаг 2 — Анализ**

- Ожидание живёт в Redis (`lab:{user_id}` с TTL = 2700 с). Постоянный аудит цикла — в PostgreSQL (`lab_cycles`).
- По истечении TTL Celery Beat отправляет push «Анализ готов».
- Анализ **не прерывается** при смене состояния (infected → zombie) и при закрытии бота; цикл доигрывается до конца.

**Шаг 3 — Результат**

- Infected: `timer_ends_at += rand(4…8 ч)`, пишется событие в `timer_history`.
- Zombie: результат засчитывается в **цепочку воскрешения** (см. §5.4); `timer_ends_at` не трогаем.
- Незабранный результат у Infected сгорает через 24 ч (отдельный TTL в Redis). У зомби результат не сгорает (он не влияет на «живой» таймер).

### 3.3 Лимиты

| Параметр | Значение | Примечание |
|---|---|---|
| Циклов в UTC-сутки (Infected) | 2 | UTC 00:00 — 23:59 |
| Циклов в UTC-сутки (Zombie, Лаба воскрешения) | 1 | Отдельный лимит, не пересекается |
| Сброс лимита | UTC 00:00 | Не зависит от времени первого цикла |
| Доп. цикл сверх лимита | платный (Points / Stars) | Параметры — в отдельном спеке экономики |

### 3.4 «Пропуск» (критично для воскрешения зомби)

- **Пропуск UTC-суток** = за эти сутки не был **начат** ни один цикл Лабы.
- Незавершённый цикл (начат, результат не забран) **не считается** пропуском.
- Тайм-аут забора результата (24 ч) не влияет на подсчёт дней подряд.

### 3.5 Взаимодействие с другими механиками

| Сценарий | Поведение |
|---|---|
| Реферал пришёл во время анализа | Бонус добавляется независимо |
| Infected → Zombie во время анализа | Цикл доигрывается, результат идёт в streak |
| Фонтан открылся во время анализа | Ивент считается независимо |
| Мутация «Лаба Ур.2» (дерево мутаций) | Задержка анализа 45 → 25 мин |

### 3.6 Acceptance Criteria — Лаборатория

- **AC-LAB-01** Кнопка «Собрать образец» недоступна, если: активный цикл, исчерпан лимит суток, либо `state ∉ {infected, zombie}`.
- **AC-LAB-02** После шага 1 в Redis пишется `lab:{user_id}` с TTL 2700 с; в БД появляется запись `lab_cycles (state="analyzing")`.
- **AC-LAB-03** Celery Beat по истечении TTL отправляет push «Анализ готов».
- **AC-LAB-04** Результат у `infected` добавляет `rand(4…8)` ч к `timer_ends_at` и пишет событие в `users.timer_history`.
- **AC-LAB-05** Результат у `zombie` инкрементирует `users.lab_revival_streak`; `timer_ends_at` не меняется.
- **AC-LAB-06** Незабранный `infected`-результат сгорает через 24 ч (отдельный TTL в Redis); `lab_cycles.state="expired"`.
- **AC-LAB-07** Лимит 2/сутки считается по UTC-дате, не по скользящему 24-часовому окну.
- **AC-LAB-08** Смена `infected → zombie` в процессе анализа не прерывает цикл; после `claim` результат идёт в streak.

## 4. Фонтан жизни

**Назначение:** коллективный глобальный ивент. Создаёт нарратив «я ↔ планета», даёт зомби путь к воскрешению.

### 4.1 Расписание

| Параметр | Значение |
|---|---|
| Частота | каждые 60 ч серверного времени (≈ 2.5 суток) |
| Объявление | push за 6 ч до открытия |
| Окно активности | 24 ч |
| Пороги наград | 75% — базовая, 100% — бонусная |

### 4.2 Формула прогресса

Каждое действие в окне добавляет «единицы» в общий бар:

| Действие | Вес | Суточный лимит |
|---|---|---|
| Подтверждённый реферал | 5 | без лимита |
| Открытие бота / чтение сообщения | 1 | 3 в сутки на игрока |
| Завершённый цикл Лаборатории | 3 | 2 в сутки (лимит Лабы) |
| Партнёрское задание | 4 | 1 в сутки |
| NPC (автоматически) | 0.5 / тик | до 25% бара, см. §4.4 |

Целевой 100% = `F × C`, где `F` — константа из env, `C = max(DAU × 0.3, 1000)`. Это удерживает «живость» бара при любом размере аудитории.

### 4.3 Награды

| Порог | Кому | Условие участия | Награда |
|---|---|---|---|
| 75% | Infected | любое действие за окно | +12 ч к таймеру |
| 75% | Zombie | любое действие за окно | право на воскрешение (см. §5.4) |
| 100% | Infected | любое действие за окно | +12 ч + случайная мутация |
| 100% | Zombie | любое действие за окно | **гарантированное** воскрешение |
| Топ-10% вкладчиков | Infected + Zombie | вклад ≥ X% бара | легендарная мутация + ранг |

**Минимум участия:** ≥ 2 единицы в бар (исключает случайные засчитывания).

### 4.4 NPC и прозрачность (продуктовое решение)

Фонтан гарантированно достигает 75%, если DAU < 500 — NPC докидывают до 25% суммарного бара. Это **внутренняя** страховка онбординга первых недель, **не анонсируется** пользователям. При DAU ≥ 500 NPC-вклад отключается. Решение пересматривается на v1.5 — когда метрики стабильны, страховку можно снять полностью.

### 4.5 Воскрешение зомби через Фонтан — детерминированно, не RNG

RNG воспринимается как несправедливость («я участвовал, но не повезло»). Детерминированность даёт ощущение контроля.

- **75%:** зомби с ≥ 2 единицами → `users.fountain_revival_ready = true`. Воскрешение срабатывает при **ближайшем** действии игрока (любой webhook-ивент от него).
- **100%:** воскрешение мгновенное, без ожидания действия.
- **< 2 единиц:** зомби не воскрешает, даже при 100% бара.

### 4.6 Acceptance Criteria — Фонтан

- **AC-FNT-01** Фонтан создаётся Celery Beat каждые 60 ч; запись в `events (type="fountain")`.
- **AC-FNT-02** За 6 ч до открытия всем активным (`active_within_7d`) уходит push.
- **AC-FNT-03** Каждое засчитанное действие пишет запись в `fountain_contributions (user_id, event_id, weight, action_type, created_at)`.
- **AC-FNT-04** Суммарный прогресс пересчитывается батчем раз в 60 с; хранится в Redis `fountain:{event_id}:progress`.
- **AC-FNT-05** При пересечении 75% воркер помечает участвующих зомби `users.fountain_revival_ready = true`.
- **AC-FNT-06** Зомби с `fountain_revival_ready = true` воскресает при ближайшем webhook-событии от него.
- **AC-FNT-07** При 100% — воскрешение всех участвовавших зомби немедленное, без ожидания действия.
- **AC-FNT-08** Игрок видит в UI свой вклад: «твой вклад: N ед. (X% бара)».
- **AC-FNT-09** NPC-вклад активен только при DAU < 500; в публичных текстах не упоминается.

## 5. Зомби-режим

**Ключевой принцип:** зомби — другой **режим игры**, а не деградированная версия Infected.

### 5.1 Переход Infected → Zombie

- Триггер: `timer_ends_at ≤ now()`, Celery Beat каждые 60 с.
- Переход **не мгновенный для игрока**: за 30 мин до дедлайна — push «Последний шанс» с кнопкой, открывающей экран передачи штамма.
- Если за 30 мин новых заражений нет — `state = "zombie"`, push с нарративом (не «ты проиграл»).

### 5.2 Правила режима

| Параметр | Infected | Zombie |
|---|---|---|
| Таймер до перехода | `timer_ends_at` активен | игнорируется; в UI — полоса некроза |
| Ручной шаринг | обычный | работает, но не продлевает таймер (его нет) |
| Хаотичное заражение | — | авто (§5.3) |
| Лаборатория | 2/сутки | 1/сутки (Лаба воскрешения) |
| Участие в Фонтане | да | да |
| Мутации (`mutation_tree`) | активны | заморожены; XP не идёт |
| Партнёрские задания | да | да (вклад в Фонтан) |
| Рейтинг | общий | отдельный зомби-рейтинг |

### 5.3 «Хаотичное заражение»

Это автоматическая механика, **не упрощённый шаринг**:

- Раз в 6 ч воркер берёт случайного Infected из цепочки зомби вверх по `infected_by_user_id`.
- С вероятностью 50% добавляет этому Infected **+1 ч** к `timer_ends_at` (и пишет событие в `timer_history`).
- Зомби видит: «Ты заразил @username — инстинктивно».
- Инфицированный источник бонуса видит событие в истории таймера как «хаотичный толчок» (без имени зомби, чтобы не раскрывать цепочку).
- Анти-абуз: лимит по одной и той же паре (infector, target) — не чаще 1 раза в 24 ч; зомби с нулевой цепочкой → механика молча пропускается.

### 5.4 Пути выхода — и разведение двух счётчиков

**Важно.** В исходной черновой формулировке «lab_revival_streak» одновременно рос и от Лабы, и от ручного шаринга. Это создавало путаницу. Разводим на два поля:

| Поле | Что считает |
|---|---|
| `users.lab_revival_streak SMALLINT` | последовательные UTC-сутки с **начатым** циклом Лабы (только для зомби) |
| `users.zombie_manual_infections SMALLINT` | уникальные заражения, сделанные зомби вручную |

| Путь | Условие | Результат при воскрешении |
|---|---|---|
| Фонтан (пассивный) | участие ≥ 2 ед. + бар ≥ 75% | `state="infected"`, `timer_ends_at = now() + 24 ч` |
| Лаба воскрешения | `lab_revival_streak ≥ 3` без пропусков UTC-суток | `state="infected"`, `timer_ends_at = now() + 48 ч` |
| Ручной шаринг | `zombie_manual_infections ≥ 5` | `state="infected"`, `timer_ends_at = now() + 24 ч` |

Все три независимы. За один «зомби-эпизод» воскресить можно **один раз**; при воскрешении оба счётчика и `fountain_revival_ready` сбрасываются.

### 5.5 Копирайт

Нарратив карточки зомби (по-русски) уже в `app/i18n/messages.py::zombie_card_*`. Тексты: короткие, без «game over», с подсказкой про «🛡 Иммунитет» / Mini App.

### 5.6 Acceptance Criteria — Зомби

- **AC-ZMB-01** Переход `infected → zombie` — только через Celery Beat, не синхронно в webhook.
- **AC-ZMB-02** За 30 мин до дедлайна игроку уходит push с кнопкой «Последний шанс» (открывает экран передачи штамма).
- **AC-ZMB-03** У зомби `timer_ends_at` не учитывается в отображении; на карточке — полоса некроза.
- **AC-ZMB-04** Хаотичное заражение запускается воркером раз в 6 ч; результат — `+1 ч` случайному Infected из цепочки с вероятностью 50%; пара (infector, target) не повторяется чаще 1 раза / 24 ч.
- **AC-ZMB-05** Ручной шаринг зомби создаёт нового Infected и инкрементирует `users.zombie_manual_infections`.
- **AC-ZMB-06** Цикл Лабы у зомби инкрементирует `users.lab_revival_streak` (не `zombie_manual_infections`).
- **AC-ZMB-07** Пропуск UTC-суток без начатого цикла Лабы сбрасывает `lab_revival_streak` в 0.
- **AC-ZMB-08** При воскрешении любым путём: `state="infected"`, `lab_revival_streak=0`, `zombie_manual_infections=0`, `fountain_revival_ready=false`; зомби-эпизод закрыт.
- **AC-ZMB-09** Зомби-рейтинг отдельный; в общих топах не смешивается.
- **AC-ZMB-10** В зомби-режиме `mutation_tree` заморожен; XP и новые мутации не начисляются.

| Фаза | Содержание |

| 0 | Конфиг таймера (48 / 72), `timer_history`, UI-блок «от чего растёт таймер» |
